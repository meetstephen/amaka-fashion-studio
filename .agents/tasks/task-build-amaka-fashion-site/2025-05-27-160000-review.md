# Luxury menswear site for Amaka Fashion Atelier

A greenfield Next.js 16 site implementing a luxury menswear brand presence for a Nigerian tailor based in Abakaliki. Four pages (Home, Collections, About, Contact), a keyword-matching AI chatbot ("Dapper"), a floating WhatsApp CTA, and Framer Motion animations throughout. The design system uses Tailwind CSS v4's `@theme` directive with custom emerald/gold/cream/black palette and Playfair Display + Inter font pairing.

**Watch for:** Contact form constructs a WhatsApp URL from raw user input without encoding, creating both a usability bug and a potential open-redirect vector (confirmed). Every page component is `"use client"`, which opts the entire site out of React Server Components and SSR metadata benefits (confirmed). The chatbot has no scroll-to-bottom on initial open for returning users and no message persistence (likely minor UX gap).

## High-level view

The entire source is client-rendered — every page carries the `"use client"` directive, meaning the metadata export in `layout.tsx` is the only server-rendered piece. This forgoes streaming, suspense boundaries, and per-page metadata that App Router was designed around.

The "Dapper" chatbot is a self-contained keyword-match engine with 20 intents, no external API calls, and no state persistence. The keyword matching is first-match-wins with no scoring, so intent ordering determines priority when multiple keywords match — and substring matching means short keywords like "hi" fire inside longer words like "shirt" and "chieftain".

The contact form is a WhatsApp deep-link generator that builds a `wa.me` URL from form fields. The URL construction uses manual `%0A` literals mixed with unencoded user input — a user typing `&` or `%` in the message field will produce a malformed URL or unintended query parameters.

The animation surface is large (every card, every section header, every page transition) but uses `viewport={{ once: true }}` consistently, avoiding replay-on-scroll jank. All variants are centralized in `animations.ts`.

## WhatsApp URL construction is unsafe

The contact form submission handler in `contact/page.tsx` builds the WhatsApp message URL by string interpolation:

```tsx
const text = `Hello Amaka Fashion Atelier!%0A%0AName: ${formData.name}%0AEmail: ${formData.email}%0APhone: ${formData.phone}%0AMessage: ${formData.message}`;
window.open(`https://wa.me/2349131272407?text=${text}`, "_blank");
```

The `%0A` sequences are manually pre-encoded newlines, but the user-supplied field values are injected raw. A message containing `#`, `&`, `%`, or non-ASCII characters will either truncate the message at the `#` (fragment identifier), inject unintended query params, or produce a double-encoded mess. There is no `encodeURIComponent` call anywhere in the file.

The fix: build the message as a plain string with real newlines, then pass it through `encodeURIComponent` once before appending to the URL. The raw interpolation also means a crafted input could construct an arbitrary `wa.me` URL suffix — WhatsApp's own URL parser limits the blast radius, but the pattern is unsound.

## Full client-rendering removes App Router benefits

Every page file begins with `"use client"`. The `layout.tsx` is the sole server component. None of the page components can export `metadata` (a server-only API), none benefit from streaming HTML, and the entire React tree ships as a client bundle. For a content-heavy marketing site, search engines receive a minimal HTML shell, and the first contentful paint depends on JS execution.

The interactive parts are limited — the collection filter, the chatbot, the contact form, and the Framer Motion animations. These could be extracted into `"use client"` islands while leaving the page shells as server components with per-page metadata exports.

## Chatbot intent matching: first-match bias

The chatbot iterates `intents` in array order and returns the first match where any keyword appears in the lowercased input:

```ts
for (const intent of intents) {
  if (intent.keywords.some((kw) => lower.includes(kw))) {
    return intent.reply;
  }
}
```

The greeting intent sits at index 0 with keywords `["hello", "hi", "hey", ...]`. Any user message that begins with "Hi, I'd like a suit" will match the greeting, never reaching the suit intent at index 2. A simple improvement: score by longest matching keyword, or check more-specific intents first (move greetings to the end of the array).

Additionally, substring matching means "hi" matches inside "shirt", "shipping", "chieftain". The keyword `"hi"` will trigger the greeting for any message containing those words. Word-boundary checks or splitting on whitespace would reduce false positives.

## Collections page dead-end affordance

Each item card in the collections grid renders a "View Details" label with an `ArrowRight` icon:

```tsx
<div className="mt-4 flex items-center gap-2 text-emerald">
  <span className="text-xs uppercase tracking-[0.15em] font-medium">
    View Details
  </span>
  <ArrowRight size={12} className="transition-transform group-hover:translate-x-1" />
</div>
```

No wrapping `<Link>`, no `onClick`, no `href`. The arrow animates on hover but clicking does nothing. Either wire it to a detail route (which doesn't exist yet) or remove the affordance and replace it with a WhatsApp inquiry link.

## Design system: emerald token shadows Tailwind's built-in scale

The `--color-emerald` token (`#006B3F`) shadows Tailwind's built-in `emerald` color scale. In Tailwind v4, custom `@theme` values take precedence, so all references to `text-emerald`, `bg-emerald` resolve to the custom value. This means any future use of Tailwind's native emerald scale (e.g., `emerald-100` for subtle backgrounds) is unavailable unless explicitly re-declared. Consider using a non-colliding name like `brand-green` or `amaka-emerald`.

## Accessibility gaps

- The mobile nav toggle button doesn't set `aria-expanded` to reflect open/closed state. Screen readers won't communicate whether the menu is visible.
- The chat window has no focus trap. When opened, keyboard focus doesn't move into the chat, and tabbing can escape behind the overlay into page content.
- The collections filter buttons don't use `aria-pressed` or a radio/tab pattern. The active state is purely visual.
- Color contrast: gold text (`#D4AF37`) on white/cream backgrounds is likely below WCAG AA for small text (estimated ~2.8:1 ratio). The gold-on-emerald combinations in the chatbot header may also fall short.

## No test suite

The chatbot's `generateReply` function and the contact form's URL construction are pure functions that would benefit from unit tests — especially given the encoding bug identified above. A handful of tests covering keyword priority, substring false positives, and URL encoding edge cases would catch regressions cheaply.
